## File Encoding

- 모든 텍스트 파일은 UTF-8 with BOM으로 읽고 저장한다.
- PowerShell에서 한글이 포함된 텍스트 파일을 읽을 때는 `Get-Content -Encoding UTF8`을 사용한다.

## Coding Conventions

- private 필드는 `_camelCase`를 사용한다. 예: `_videoPlayer`, `_isLoading`
- enum은 주로 `E` 접두사를 사용한다. 예: `EBattlePopupType`, `EAppState`
- 상수(`const`)는 `UPPER_SNAKE_CASE`를 사용한다. 예: `CATEGORY_NAME`, `KEY_SORT_OPTION`
- `bool` 조건은 `== true` / `== false` 비교 대신 `if (condition)` / `if (!condition)` 형태를 사용한다.

## 코딩 가이드라인(Coding Guidelines)

**트레이드오프(Tradeoff):** 정확성, 성능, 신중함을 우선하되 작업의 위험도와 규모에 맞게 판단한다.

### 1. 코딩 전에 생각하기(Think Before Coding)

**추측을 숨기지 말고, 중요한 트레이드오프를 드러낸다.**

- 구현 방향, 권한, 변경 범위에 실질적인 영향을 주는 불확실성이 있으면 무엇이 불명확한지 설명하고 최소한의 질문으로 확인한다.
- 그 외에는 가정을 명시하고 안전한 범위에서 진행한다.
- 여러 설계 선택지가 결과에 큰 영향을 주면 대안을 비교해 제시한다. 더 단순한 접근법이 있으면 함께 제안한다.
- 사용자가 지정한 방향보다 설계적으로 더 나은 대안이 있고 그 차이가 중요하면 이유와 함께 제안한다.

### 2. 단순함 우선(Simplicity First)

**요청을 해결하는 최소한의 코드만 작성하고, 변경 범위를 필요한 곳으로 제한한다.**

- 명시적으로 요청되지 않은 기능, 추측성 코드, 불필요한 추상화·인터페이스·제네릭 클래스·설정 가능성을 추가하지 않는다.
- 정상 동작하는 코드를 인접했다는 이유만으로 리팩터링하거나, 관련 없는 코드·주석·포맷팅을 임의로 개선하지 않는다.
- 기존 프로젝트의 스타일과 컨벤션을 따른다.
- 내 변경으로 사용되지 않게 된 import, 변수, 함수만 정리한다. 기존의 관련 없는 데드 코드나 버그는 언급만 하고 요청 없이 수정하지 않는다.
- 모든 변경은 사용자의 구체적인 요청을 직접 지원해야 한다.
- 논리적으로 발생할 수 없는 시나리오를 위한 예외 처리나 추측성 분기를 추가하지 않는다.
- 구현 후에는 변경된 동작을 직접 검증하는 테스트를 우선한다. 테스트를 실행할 수 없으면 이유와 다음 확인 방법을 명시한다.
- 같은 동작을 더 단순하게 표현할 수 있다면 단순화한다. 불필요한 복잡성이 의심되면 먼저 더 작은 설계를 검토한다.

### 3. 권한 경계와 검증(Autonomy and Validation)

- 답변, 설명, 검토, 진단, 계획 요청에서는 관련 자료를 조사하고 결과만 보고한다. 구현은 명시적으로 요청된 경우에만 한다.
- 변경, 구현, 수정 요청에서는 범위 내 로컬 변경과 관련된 비파괴적 검증까지 진행한다.
- 외부 쓰기, 파괴적 작업, 비용 발생, 또는 실질적인 범위 확장은 먼저 확인한다.
- 변경 뒤에는 가능한 가장 관련성 높은 검증을 수행한다. 검증할 수 없으면 이유와 다음 확인 방법을 명시한다.

### 4. 목표 기반 실행과 계획(Goal-Driven Execution)

**동작, 인터페이스, 데이터에 영향을 주는 다단계 작업은 구현 전에 간단한 Markdown 계획과 검증 방법을 제시한다.**

- 작업을 검증 가능한 목표로 바꾼다. 예: 유효성 검사는 잘못된 입력 테스트로, 버그 수정은 재현 방법과 수정 검증으로 정의한다.
- 계획은 필요한 단계만 포함하며, 각 단계에 확인 방법을 연결한다.

  1. [단계] → 검증: [확인 방법]
  2. [단계] → 검증: [확인 방법]

### 5. 영향도 분석(Impact Analysis)

**공개 API, 공통 유틸리티, 시그니처, 반환값, 핵심 상태를 변경할 때 영향 범위를 먼저 확인한다.**

- 호출자와 참조를 확인하고, 예상되는 호환성 영향을 명시한다.
- 호출자가 적은 경우(≤ 10)는 모두 확인하고 필요한 변경을 반영한다.
- 호출자가 많은 공통 코드라면 기존 시그니처와 동작을 유지하고 내부 구현만 수정하는 것을 원칙으로 한다. 추가 기능은 오버로드를 우선한다.
- 광범위한 호환성 변경이 불가피하면 임의로 진행하지 말고 영향과 대안을 보고한 뒤 승인을 기다린다.
